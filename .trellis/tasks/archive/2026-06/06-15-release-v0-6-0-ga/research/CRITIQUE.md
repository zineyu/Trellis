# Adversarial Critique — v0.6.0 GA deliverables

- **Date**: 2026-06-15
- **Scope**: 4 deliverable groups — GA changelog (en + zh), `0.6.0.json` manifest, bundled `trellis-channel` skill, `trellis-meta` rewrite
- **Inputs**: every file under `.trellis/tasks/06-15-release-v0-6-0-ga/research/` + `prd.md` + `design.md` + `implement.md`

Severity legend: **CRITICAL** = blocks GA / breaks contract, **SHOULD-FIX** = visible inconsistency, **NICE-TO-HAVE** = polish.

---

## A. `trellis-meta` rewrite (skill + 2 new references)

### A1. CRITICAL — `bundled-skills.md` explicitly denies the `trellis-channel` skill exists

File: `research/trellis-meta-rewrite/references/local-architecture/bundled-skills.md` lines 28-29:
> "Older docs mention `trellis-channel` as a candidate bundled skill. **It is not currently shipped under `bundled-skills/` in this release; the `trellis channel` runtime ships as CLI commands, not as a skill directory.**"

This directly contradicts prd.md line 20 ("Ship new `trellis-channel` bundled capability skill"), prd.md success criterion #10 ("`packages/cli/src/templates/common/bundled-skills/trellis-channel/SKILL.md` exists"), and implement.md A0.1. The "Current Bundled Skills (v0.6.0)" table (lines 22-27) lists only `trellis-meta` / `trellis-session-insight` / `trellis-spec-bootstrap` — must add a fourth row for `trellis-channel`, and delete the contradiction paragraph entirely.

### A2. CRITICAL — `trellis-meta/SKILL.md` "Current Rules" omits the new `trellis-channel` bundled skill

File: `research/trellis-meta-rewrite/SKILL.md` line 73:
> "Bundled multi-file skills (`trellis-meta`, `trellis-spec-bootstrap`, `trellis-session-insight`) are auto-dispatched to every platform skill root by `getBundledSkillTemplates()` …"

Line 82 ("Do not put team-private project rules into any public bundled skill…") lists the same three. Both must include `trellis-channel` so AI agents reading the meta skill know it exists and is auto-dispatched.

### A3. SHOULD-FIX — `trellis-meta/SKILL.md` frontmatter description never mentions `trellis-channel` or `trellis mem` as bundled skills

Line 3 description ends "…or cross-session memory (`trellis mem`) generated by `trellis init`." — it lists `trellis channel` (the runtime) and `trellis mem` (the CLI) as architecture surfaces, but says nothing about the AI-facing bundled skills that wrap them. Trigger surface should add `trellis-channel`, `trellis-session-insight`, `trellis-spec-bootstrap` and "bundled-skill auto-dispatch" so AI dispatch matches the right phrase.

### A4. SHOULD-FIX — `multi-agent-channel.md` defers all command syntax to "the bundled `trellis-channel` skill" without naming `.trellis/agents/` per-platform sub-agent vs. channel-runtime agent distinction

File: `research/trellis-meta-rewrite/references/local-architecture/multi-agent-channel.md` lines 60-65 are good, but the reference never points back to `platform-files/agents.md` which is supposed to call out the distinction (per `trellis-meta-drift.md` items 11). Add one bullet under "Relationship To Other Local Layers" stating "Editing `.claude/agents/trellis-implement.md` (and peers) does NOT change channel-runtime worker behavior — edit `.trellis/agents/<name>.md`." This rule already exists in SKILL.md line 84 but is missing from the dedicated reference file.

## B. `trellis-channel` bundled skill (SKILL.md + 5 references)

### B1. SHOULD-FIX — Porting plan asked for thread-key scrubs in `forum.md`; they were not applied

`research/trellis-channel-porting-plan.md` §3.2 said to replace illustrative thread keys with neutral examples:
> "Replace `channel-threads-core-beta13` and `release-ci-only-publishing` with generic placeholders (`release-2026-q1` and `runtime-event-schema-change`)"

Actual `research/bundled-trellis-channel/references/forum.md` lines 225-226 still ship:
> "Use stable, descriptive thread keys (e.g. `channel-threads-0-6`, `release-ci-only-publishing`)"

`release-ci-only-publishing` is the exact maintainer-leftover string the porting plan flagged for removal. Either accept the plan's recommended replacements or document why the leftover stays.

### B2. SHOULD-FIX — `command-reference.md` carries a "Drift vs global skill" footer it should not have in the bundled copy

`research/bundled-trellis-channel/references/command-reference.md` lines 5-7 reference "the global skill (`~/.claude/skills/trellis-channel/references/command-reference.md`)" and promise a "Drift vs global skill" section at the bottom. The bundled skill ships to every user — they have no `~/.claude/skills/trellis-channel/`. Either remove the front-matter sentence and the closing drift section, or rewrite as "drift vs. previously-shipped revisions of this file."

### B3. NICE-TO-HAVE — SKILL.md description is longer than the porting plan recommended

Porting plan §5: one sentence. Actual draft frontmatter line 3 has ~70 words including a comma-separated list of trigger phrases. Long descriptions can dilute the auto-trigger match score on some platforms (mintlify-style frontmatter parsing truncates after ~200 chars on Cursor). Consider shortening to the porting plan's one-line form and moving the verbatim trigger phrases into the body.

## C. `0.6.0.json` migration manifest

### C1. SHOULD-FIX — `changelog` "Bundled skills" section omits `trellis-channel`

File: `research/0.6.0-manifest-draft.json` line 6, "Bundled skills" subsection mentions `trellis-spec-bootstrap` and `trellis-session-insight`, then the third line is vague:
> "- Channel runtime bundled skills auto-discovered by AI agents on every platform."

Should be explicit and parallel to the other two: `- **trellis-channel** (new in v0.6.0 GA cycle): teaches AI agents when to reach for `trellis channel` for multi-agent brainstorm, peer review, forum/thread boards, and dispatcher-wait patterns.` This is the manifest output users see during `trellis update`; vague phrasing here makes it harder to grep.

### C2. NICE-TO-HAVE — `changelog` string is ~5 KB; v0.5.0 precedent + the json-template doc target 3-4 KB

`research/0.6.0-json-template.md` §4 says "Total target: 3–4 KB, mirroring v0.5.0's ~3 KB shape." The draft string runs longer than that, mostly because the **Updater** and **Workflow + planning** sections each accumulated 5-7 bullets. Could be trimmed without losing content the user needs at `trellis update` time (e.g. "session-start version-update hint" is already in the per-beta changelog).

### C3. NICE-TO-HAVE — `notes` field doesn't tell users about the OpenCode degradation

Manifest line 8 covers 0.5.x users, 0.6.0-prerelease users, Codex users, and the channel runtime opt-in — but never mentions that `trellis mem` returns empty on OpenCode 1.2+. EN/ZH changelog drafts both have a dedicated `<Note>` for this. The manifest's terminal-rendered `notes` should at least have one line so OpenCode users on `trellis update` see the warning inline.

## D. GA changelog EN + ZH drafts

### D1. CRITICAL — EN and ZH drafts are NOT 1:1 mirrored (violates design.md §"GA changelog synthesis approach")

design.md line 69: "EN and ZH 1:1 mirrored, prose translated, code blocks and tables identical." Concrete divergences:

- **EN has 3 `<Note>` blocks** (Codex / New platforms / OpenCode). **ZH has 2** (Codex / Reasonix-only — Pi is folded into a later H2, OpenCode is only in the `<Warning>`). en-draft.md lines 12-33 vs zh-draft.md lines 12-25.
- **EN `## Bug Fixes`** (lines 167-179) has ONE entry (rc.0 Exa MCP fix). **ZH `## Bug 修复 & 内部清理 —— 头条`** (lines 204-216) is a 7-row TABLE covering Exa MCP + Codex toml prelude duplicate + archive auto-commit + slug collision + Cursor sessionStart + OpenCode shell dialect + Hook timeout + session-start version hint. Design.md §6 says "only the GA-cut-window ones (the rc.0 exa MCP fix)" — **ZH is over-scoped**, EN is correct. Trim ZH to match.
- **Line counts**: EN ~208 lines, ZH ~241 lines. Target was ~190 lines (v0.5.0 precedent). Both run long but the gap between them is the 1:1 violation.

### D2. SHOULD-FIX — `<Tip>` content differs substantively between EN and ZH

- EN (line 9) leads with "Multi-agent collaboration is now a first-class primitive" and ends pointing at the **`check`/`implement` agent definitions auto-install** angle.
- ZH (line 9) leads with "`trellis channel` 是 v0.6 的旗舰能力" and ends pointing at **`--type threads` / `--type forum` persistent boards**.

Same theme, different lead. Per design.md "prose translated," the two should converge on one lead. Recommend the EN lead (agent definitions angle is the more user-actionable shift per v0.5.0-changelog-structure.md §3 "if you only read one thing" rule).

### D3. SHOULD-FIX — Missing `## RC stabilization` section (precedent calls it mandatory for GA)

`research/v0.5.0-changelog-structure.md` §6 lists "`## RC stabilization (rc.0 → rc.N)` — mandatory for GA". v0.5.0 GA had this section. v0.6.0 only shipped rc.0 (no rc.1+), so the table would have one row, but the precedent expects the section to exist for GA-changelog ritual reasons (it signals "this is what GA bundles up"). Both EN and ZH drafts omit it.

Minimal version: a single H2 with one line — "GA = rc.0 with zero src/ changes; no rc.1 cut. The breaking work happened at `0.6.0-beta.0`."

### D4. SHOULD-FIX — Missing dedicated `## Platform coverage` H2 section

v0.5.0 precedent has `## Platform coverage` as its second H2 (per `v0.5.0-changelog-structure.md` §6). v0.6.0 EN draft scatters platform info: Reasonix + Pi are in `<Note>`, Codex / Cursor / OpenCode / Copilot are in the manifest's "Platform coverage" section but not in the .mdx draft as a dedicated H2. ZH has it implicitly across `## 平台新增` (line 124) but the section name doesn't grep-match v0.5.0 style. Recommend adding `## Platform coverage` H2 to both, lifting the Reasonix + Pi + Codex / Cursor / OpenCode / Copilot bullets from the manifest's `changelog` field.

### D5. SHOULD-FIX — ZH `<Tip>` anchor link target `#多-agent-协作-trellis-channel` is uncertain under Mintlify anchor generation

Mintlify generates anchors from heading text via a deterministic slugifier that handles CJK inconsistently. ZH H2 is `## 多 agent 协作 (\`trellis channel\`)` — the parens + backticks + mixed CJK/ASCII may produce an anchor different from `#多-agent-协作-trellis-channel`. EN equivalent `#multi-agent-collaboration` is safe. Either test the ZH anchor in a staging preview, or rewrite the ZH H2 to a simpler form before shipping (e.g. `## 多 agent 协作`).

### D6. NICE-TO-HAVE — EN draft omits `84 unit tests / 81.89% coverage` detail that the manifest draft and v06-cycle-synthesis.md both call out

EN line 79-91 covers `trellis mem` but doesn't carry the coverage numbers that the manifest's `changelog` field includes. v0.5.0 precedent did cite per-feature test counts in similar sections. Add one parenthetical to the EN "Memory (`trellis mem`)" intro paragraph.

### D7. NICE-TO-HAVE — EN `## Upgrade` block doesn't mention the `0.6.0-beta.23` `rename-dir` migration

EN line 187-201 covers the breaking-gate logic but the manifest's `notes` field (line 8 of the draft) and v06-cycle-synthesis both call out the `rename-dir` migration (renames installed `trellis-spec-bootstarp/` → `trellis-spec-bootstrap/` across 13 platform roots). Users running `update --migrate` from 0.5.x will see this happen; the GA changelog should warn them.
