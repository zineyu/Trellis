## 1. Frontmatter Format

```mdx
---
title: 'v0.5.0'
description: '2026-05-06'
---
```

**Pattern**: Single-quoted strings. `title` = version tag (`vX.Y.Z`). `description` = release date as `YYYY-MM-DD` (NOT a sentence — date doubles as description).

**v0.6.0 adaptation**:
```mdx
---
title: 'v0.6.0'
description: '2026-06-15'
---
```
(Use actual GA ship date in `description`.)

## 2. Opening Paragraph

```
Stable promotion of `0.5.0-rc.6` with no new src/ changes. v0.5.0 is a breaking release from 0.4.x — skill-first architecture, 7 platforms upgraded to agent-capable, `workflow.md` as the single source of truth for the workflow.
```

**Pattern** — single paragraph immediately after frontmatter, two sentences:
1. **Promotion sentence**: "Stable promotion of `X.Y.Z-rc.N` with no new src/ changes." Names which RC is being promoted; confirms no code drift between final RC and GA.
2. **Headline sentence**: `vX.Y.Z is a {breaking|minor|patch} release from {prev}.x — <3 most important shifts joined by commas>`. Breaking marker is the word "breaking" inline (no special tag). Three highlights at most, comma-separated, em-dashed off the version statement.

**v0.6.0 adaptation**: Identify which RC is the GA basis (likely `0.6.0-rc.N`). State whether it's breaking from `0.5.x` or a minor bump. Pull 2-3 headline shifts from the rc-series — candidates likely include channel/multi-agent collaboration, brainstorm iteration policy, the docs-site bilingual flow, or whatever the rc.0 → rc.N stream actually delivered.

## 3. `<Tip>` Usage

```mdx
<Tip>
**`/start` is no longer a required entry point.** Just describe what you want in natural language — you're already in the Trellis workflow. `/continue <what you want to do>` works as an explicit kickoff if you want it. If you'd rather manually start a session before chatting, `/trellis:continue` now serves as the kickoff command in place of `/start`. See the "[/continue command](#/continue-command)" section below.
</Tip>
```

**Pattern**: Exactly ONE `<Tip>` in the file, placed immediately after the opening paragraph. Content = **the single most actionable user-facing behavior shift** in this release. Format:
- Opens with a bolded one-sentence headline (`**...**.`)
- 2-4 follow-up sentences explaining the new mental model
- Ends with an in-page link to the deep-dive section (`[/continue command](#/continue-command)`)

**Rule**: Reserve `<Tip>` for the "if you only read one thing, read this" behavioral shift — usually the change most likely to confuse existing users on day 1.

**v0.6.0 adaptation**: Pick the single highest-leverage UX shift in 0.6.0 (e.g. channel-based collaboration becoming default, or whatever's mostly likely to surprise a 0.5.x user). Bold-lead, explain, link to a `## Section` later in the file. Only one `<Tip>`.

## 4. `<Note>` Usage

```mdx
<Note>
**Codex users — beta-feedback fix in 0.5.0:**

- **`multi_agent_v2` default-on (rc.5)** — `.codex/config.toml` template writes the feature block instead of leaving it commented. The `min_wait_timeout_ms = 480000` (8 min) `wait()` floor stops the parent thread from busy-polling subagent status. **Requires Codex CLI ≥ v0.128.0** — older Codex will fail with `Error loading config.toml: data did not match any variant of untagged enum FeatureToml in features.multi_agent_v2`.

</Note>
```

**Pattern**: `<Note>` is for **per-platform upgrade caveats that are actionable by Trellis users** (fixes/changes shipped by Trellis that require user action or awareness):
- Bolded headline `**<Platform> users — <topic>:**`
- Bullet list of caveats, each starting with a `**<Feature/change>**` lead
- Each bullet includes: what changed → why it matters → version constraint or failure mode (verbatim error text when available)

**Rule**: One `<Note>` per affected platform; each bullet must be actionable (version pin, config change, user can do something about it).

**v0.6.0 adaptation**: Use `<Note>` for any 0.6.0 platform-specific upgrade requirements (e.g. minimum CLI versions for Claude Code / Codex / Cursor, new env vars, config.toml regenerations). If multiple platforms have caveats, use multiple `<Note>` blocks back-to-back. If none, omit.

## 5. `<Warning>` Usage

```mdx
<Warning>
**Known Codex upstream issues (not fixable from Trellis):**

- **Hook context rendered in terminal ([#191](https://github.com/mindfold-ai/Trellis/issues/191))** — Codex prints SessionStart hook context to the terminal on every turn. No toggle to suppress it for now (the Codex desktop app avoids this).
- **Sub-agent startup hangs on a slow / failing MCP server** — sub-agent init can stall waiting on an MCP that never returns. Reported since Codex `multi_agent_v1`, still present in `v2`.

</Warning>
```

**Pattern**: `<Warning>` is for **known issues OUT of Trellis's control** — upstream bugs in the host CLI/IDE that Trellis cannot fix:
- Bolded headline `**Known <platform> upstream issues (not fixable from Trellis):**`
- Bullet list, each `**<symptom>** ([#issue-link])** — <one-sentence cause/workaround>`
- Each bullet links a GitHub issue for tracking when available
- Explicitly distinguishes "this is upstream, here's the issue tracker" from `<Note>` which is "we changed something, here's what to do"

**v0.6.0 adaptation**: Audit `mindfold-ai/Trellis` issues filtering for `upstream` or `wontfix-by-us` tags on platforms shipped in 0.6.0. List each as a bullet with the issue link. If none remain or all were resolved, omit the `<Warning>` block.

## 6. H2 Section Structure

Order of `##` sections in v0.5.0:
1. `## Architecture` — biggest conceptual shifts (with `###` sub-sections per shift)
2. `## Platform coverage` — what platforms got added/upgraded/dropped (with `###` sub-sections)
3. `## SessionStart payload restructure` — specific perf/size table (no sub-sections, just a table)
4. `## Migration & update flow` — `trellis update` behavioral changes (single big before/after table)
5. `## Cleanup` — removed features (intro line + 1 table)
6. `## RC stabilization (rc.0 → rc.6)` — per-RC delta table (version | change)
7. `## Upgrade` — install/upgrade commands (always last)

**Patterns**:
- Order is roughly: most conceptual → most concrete → operational.
- Each section uses tables liberally (before/after comparisons, version deltas, platform matrices). Tables are the dominant content form.
- `###` sub-sections used only inside the first two `##` sections; later sections are flat.
- The "RC stabilization" section is unique to a GA changelog — it's a version | change ledger summarizing all the prereleases the GA rolls up.
- "Upgrade" is always the closing section.

**v0.6.0 adaptation — recommended skeleton**:
```
## <theme of biggest 0.6.0 shift>           ← analog of "Architecture"
  ### <sub-shift 1>
  ### <sub-shift 2>
## Platform coverage                          ← if any platform churn in 0.6.0
## <any feature-specific deep-dive sections, one per major change>
## Migration & update flow                    ← if behavior changed
## Cleanup                                    ← if anything removed
## RC stabilization (rc.0 → rc.N)             ← mandatory for GA
## Upgrade                                    ← always last
```

Tailor the early `##` sections to whatever themes dominate the 0.6.0 rc-series commit log. Drop sections that don't apply (e.g. no `## Cleanup` if nothing was removed).

## 7. Install / Upgrade Section Format

```mdx
## Upgrade

From 0.4.x:

```bash
trellis update --migrate
```

The `--migrate` flag is REQUIRED — the breaking-change gate from `0.5.0-beta.0` fires when traversing the migration chain. 138-entry `safe-file-delete` is hash-verified; local customizations are preserved with a warning. Per-prompt `reason` field explains version-specific nuances inline.

From any 0.5.0 prerelease (`beta.X` / `rc.X`):

```bash
trellis update
```

Plain `trellis update` — clean version bump, no flag needed.

Install:

```bash
npm install -g @mindfoldhq/trellis
```
```

**Pattern**:
1. `## Upgrade` heading.
2. **One subsection per upgrade path**, label-then-code-block style:
   - `From <prev-major>.x:` → fenced `bash` command → 1-paragraph explanation of why the flags are what they are, including a verbatim error mention if missing the flag would break things.
   - `From any <current-major> prerelease (\`beta.X\` / \`rc.X\`):` → fenced `bash` command → 1-line explanation.
3. Final `Install:` label (no `##` heading) → fenced `bash` install command (`npm install -g @mindfoldhq/trellis`).

**v0.6.0 adaptation**:
```mdx
## Upgrade

From 0.5.x:

```bash
trellis update [--migrate if breaking]
```

<explanation of whether --migrate is needed for 0.5.x → 0.6.0 and why>

From any 0.6.0 prerelease (`beta.X` / `rc.X`):

```bash
trellis update
```

Plain `trellis update` — clean version bump, no flag needed.

Install:

```bash
npm install -g @mindfoldhq/trellis
```
```

Decide `--migrate` requirement based on whether `src/migrations/` contains any 0.6.0 rename/delete entries. If yes, REQUIRED phrasing matches v0.5.0 precedent. If no, drop the `--migrate` and explain.

## MDX Gotchas (confirmed from v0.5.0 source)

1. **Closing tags at column 0** — `</Tip>`, `</Note>`, `</Warning>` all start at column 0 (verified lines 10, 17, 25). Mintlify/MDX parser is sensitive to leading whitespace on closing tags inside callouts; indented closers can break rendering or leak markdown into JSX context.

2. **Blank line between bullet list and closing tag** — inside `<Note>` and `<Warning>`, there is a blank line after the last bullet, then the closing tag (verified line 16 → 17 and line 24 → 25). Skipping the blank line risks the parser merging the closer into the last list item.

3. **Blank line between opening tag and content** — `<Note>` opens, then immediately followed by content on the next line (line 12 → 13). Same for `<Warning>` (19 → 20). No blank line needed *after* the opening tag, only before the close.

4. **Inline markdown inside callouts works** — bold (`**...**`), inline code (`` `...` ``), links (`[text](url)`), and bullet lists all render correctly inside `<Tip>`/`<Note>`/`<Warning>`.

5. **In-page anchor links use `#section-name`** — line 9 references `[/continue command](#/continue-command)` linking to the `### /continue command` heading (line 48). Mintlify generates anchors from heading text; the literal `/` in the heading is preserved in the anchor.

6. **Tables use standard GFM pipe syntax** — no MDX-specific table component. Column alignment via `:---` / `---:` not used; default left-alignment everywhere.

7. **Code fences inside callouts** — v0.5.0 doesn't use fenced code blocks inside `<Tip>`/`<Note>`/`<Warning>` (only inline code). If 0.6.0 needs fenced blocks inside a callout, test it — some MDX parsers break on indentation inside JSX children.

8. **No nested lists deeper than 1 level** — every bullet list in v0.5.0 is flat (no sub-bullets). Tables are used in place of any 2-level hierarchy. Recommendation: follow the same rule for v0.6.0.

9. **Em-dashes (`—`) and en-dashes are used liberally** — these are literal Unicode characters in the source, not `--` markdown shortcuts. Preserve the style.

10. **No frontmatter `keywords` / `sidebar_title` / `description` overrides** — only `title` and `description` (which is the date). Mintlify uses `title` for sidebar and `description` for SEO meta.

## Caveats / Not Found

- Mintlify renders `<Tip>`/`<Note>`/`<Warning>` as colored callouts, but the docs-site config wasn't read.
