# Research: `0.6.0.json` migration manifest template

- **Query**: Read `0.5.0.json` and produce a field-by-field template for the v0.6.0 GA manifest, with decisions, rationale, and concrete changelog/notes strings.
- **Scope**: internal (manifest precedent + 0.6 cycle history)
- **Date**: 2026-06-15

## 1. Complete `0.5.0.json` structure (verbatim baseline)

Source: `packages/cli/src/migrations/manifests/0.5.0.json`

```json
{
  "version": "0.5.0",
  "description": "v0.5.0 stable — skill-first architecture, 7 platforms agent-capable, workflow.md as single source of truth. Stable promotion of rc.6 with no new src/ changes.",
  "breaking": false,
  "recommendMigrate": true,
  "changelog": "**v0.5.0 stable.** Stable promotion of rc.6 — no new src/ changes since rc.6.\n\n**Heads-up:** `/start` is no longer a required entry point. … (≈3 KB of bold-prefixed sections: Cycle headlines / Codex notes / Architecture / Multi-task & onboarding / Platform coverage / SessionStart payload / Migration & update flow / Cleanup / RC stabilization)",
  "migrations": [],
  "notes": "**Stable release.** Promoted from rc.6 with no new src/ changes. … Users on 0.4.x: run `trellis update --migrate` (the `--migrate` flag is REQUIRED — the breaking-change gate from `0.5.0-beta.0` fires when traversing the migration chain). … Users on any 0.5.0 prerelease (beta.X / rc.X): plain `trellis update` works as a clean version bump. \n\nInstall: `npm install -g @mindfoldhq/trellis`"
}
```

Six top-level fields total: `version`, `description`, `breaking`, `recommendMigrate`, `changelog`, `migrations`, `notes`. (`migrationGuide` is omitted; it is only required when `breaking && recommendMigrate`. Some intermediate manifests also carry `configSectionsAdded` — not needed here because GA = rc.0 with zero src changes.)

## 2. Field-by-field decisions for `0.6.0.json`

| Field | Type | 0.5.0 value | 0.6.0 value | Why |
|---|---|---|---|---|
| `version` | string | `"0.5.0"` | `"0.6.0"` | Matches the published npm version tag and `package.json#version`. The release-preflight `check-versions` step compares this string against `packages/cli/package.json`. |
| `description` | string (1 line, ≤ ~200 chars) | "v0.5.0 stable — skill-first architecture, 7 platforms agent-capable, workflow.md as single source of truth. Stable promotion of rc.6 with no new src/ changes." | "v0.6.0 stable — multi-agent channel runtime, `trellis mem`, `@mindfoldhq/trellis-core` SDK extraction, 15-platform coverage (incl. Reasonix + Pi). Stable promotion of rc.0 with no new src/ changes." | One-sentence elevator pitch. Surface (a) the cycle's defining capabilities, (b) the platform-count headline, (c) the rc.0-equivalence framing that mirrors v0.5.0. |
| `breaking` | boolean | `false` | `false` | rc.0 → GA is zero source change. The breaking work happened at `0.6.0-beta.0.json` and is already absorbed by users who upgraded through the betas. |
| `recommendMigrate` | boolean | `true` | `true` | **Counter-intuitive but correct.** Mirrors v0.5.0 precedent exactly. A user on 0.5.x walking the chain to 0.6.0 will traverse `0.6.0-beta.0.json` (which IS breaking). Setting this flag on the GA manifest is the project's way of telling `trellis update` to print "you should pass `--migrate`" guidance even though THIS specific entry has no migrations of its own. |
| `migrations` | array | `[]` | `[]` | Nothing to do at this exact rc.0 → GA step. The rename/delete chain lives in `0.6.0-beta.0.json` (and the `rename-dir trellis-spec-bootstarp → trellis-spec-bootstrap` lives in `0.6.0-beta.23.json`). Empty array preserves manifest-continuity check without re-applying historical migrations. |
| `migrationGuide` | string | (omitted) | (omitted) | Only required when `breaking && recommendMigrate`. Here `breaking: false`. |
| `configSectionsAdded` | array | (omitted) | (omitted) | rc.0 → GA adds no new `.trellis/config.yaml` sections. (`channel.worker_guard` was added in beta.18 and is already absorbed.) |
| `changelog` | string (multi-section, ~3-4 KB) | See §1 | See §4 below | Rendered to terminal during `trellis update`. Must read well in a ~100-col fixed-width window, no MDX, no HTML, just `\n` line breaks and `**bold**`. |
| `notes` | string (multi-paragraph) | See §1 | See §5 below | Renders as the "next step" hint after `trellis update` finishes traversing this manifest. Must explicitly disambiguate user cohorts (0.5.x vs 0.6.0-prerelease) on how to upgrade. |

### Rationale anchors

- **Why `recommendMigrate: true` despite empty `migrations`**: re-read `0.5.0.json` line 5 — `true` even though line 7 is `"migrations": []`. The project treats the flag as "any user reaching this manifest from older terrain should pass `--migrate`," not "this manifest has work to do." Setting it `false` would let a 0.5.x user run plain `trellis update`, hit the breaking gate at `0.6.0-beta.0`, and exit-1 mid-traverse — defeating the purpose of the GA entry's continuity. (Design.md §"Manifest field decisions" footnote confirms this is the deliberate read.)
- **Why no `migrationGuide`**: the loader logic only renders this when both `breaking` and `recommendMigrate` are true. `false && true` → not shown.
- **Why `description` calls out platform count "15"**: beta.23 added Reasonix as the 15th platform. Naming the number tracks v0.5.0's precedent of "7 platforms agent-capable" callout in its own description. Headline numbers help users skim manifest descriptions during chain traversal.
- **Why include `trellis-core` in description**: SDK extraction is the v0.6 cycle's only dual-package shift; first GA where two packages publish in lockstep. v0.5.0's description called out "skill-first architecture" — analogous "this is the big thing" beat.

## 4. Recommended `changelog` string structure

**Total target**: 3–4 KB, mirroring v0.5.0's ~3 KB shape. Opens with one framing line and one heads-up paragraph (like v0.5.0 led with the `/start` → `/continue` heads-up). Then bold-prefixed section headers in a single flat list — no nested headers, no markdown lists deeper than one level, no MDX.

Recommended section order (each section is `**Section name**\n- bullet\n- bullet\n\n`):

```
**v0.6.0 stable.** Stable promotion of rc.0 — no new src/ changes since rc.0.

**Heads-up:** Trellis now ships a multi-agent channel runtime (`trellis channel`) and a memory subsystem (`trellis mem`). You don't have to use them — the single-agent workflow you know from 0.5 still works exactly the same. But both are available out of the box and auto-discovered through the bundled `trellis-channel` and `trellis-session-insight` skills, so AI agents on every supported platform will reach for them when the task fits.

Cycle headlines since 0.5.0:

**Channel runtime**
- `trellis channel` subsystem: durable thread / forum channels, project/global scope, linked context, channel `description`, cursor reads, cross-channel watchers, inbox & delivery policies. Replaces ad-hoc multi-agent coordination with a real bus.
- Worker lifecycle: `spawn` / `kill` / `prune` / `wait` / `run`, plus turn-event recording (`interrupt_requested`, `interrupted`, `done`, `error`). Workers survive turn-level errors and `fs.watch` outages.
- OOM guard ON by default: `channel.worker_guard.idle_timeout: 5m` self-terminates idle workers; `channel.worker_guard.max_live_workers: 6` caps spawn-time live workers per project/scope. Mid-turn workers are never killed by idle cleanup. Opt out via `--idle-timeout 0` / `--max-live-workers 0` or the config block.
- Agent defs shipped: `trellis init` and `trellis update` now write `.trellis/agents/{check,implement}.md`, so switching to a channel-driven workflow no longer fails with "Agent 'check' not found" (#323).

**Memory**
- `trellis mem` searches past Claude Code / Codex / OpenCode sessions by keyword, extracts surrounding turns, and dumps full conversations. Five subcommands: `projects` / `list` / `search` / `context` / `extract`. Strips hook injections, AGENTS.md preambles, and tool-call noise so search hits reflect real dialogue. Handles compaction.
- `@mindfoldhq/trellis-core/mem` exposes the same primitives as a library: session listing, search, context extraction, project aggregation, provider adapters. Reusable from other tools.
- Bundled `trellis-session-insight` skill teaches AI agents on every supported platform when to reach for `trellis mem` instead of asking the user to repeat past context.

**SDK extraction**
- `@mindfoldhq/trellis-core` is now its own npm package, published in lockstep with `@mindfoldhq/trellis`. CLI depends on a pinned core version; `bump-versions.js` rewrites `workspace:*` → exact version at release time.
- v0.6.0 is the first GA promote to ship both packages simultaneously. Downstream tools can depend on `@mindfoldhq/trellis-core` directly without pulling the full CLI.

**Platform coverage**
- 15 platforms total. New: **Reasonix** (DeepSeek-Reasonix) via `trellis init --reasonix`; subagent skills carry `runAs: subagent` frontmatter so Reasonix spawns them as isolated subagent loops (#301).
- Pi Agent extension matured: `subagent` tool registration carries a `promptSnippet` with the `Active task:` dispatch protocol; `<workflow-state>` and `<session-overview>` injected on every `input` and `before_agent_start` event (#249). Project-level override prevents globally-installed `npm:pi-subagents` from injecting extensions / skills / prompts / themes into the current Trellis project (#246).
- Codex dispatch defaults to `inline` (was `sub-agent`). `codex-inline` / `codex-sub-agent` virtual platforms drive correct per-mode `workflow.md` branches. The `<codex-mode>` banner in `UserPromptSubmit` tells Codex which mode it is in.
- OpenCode `TRELLIS_CONTEXT_ID` now injected via the shell dialect that parses the command; `env … TRELLIS_CONTEXT_ID=…` prefixes are recognized. Sub-agent context isolated; active task resolves from session context / `Active task:` hints / single-session fallback.
- GitHub Copilot SessionStart no longer carries the stale `systemMessage`; `hookSpecificOutput.additionalContext` is the supported path.

**Updater**
- `trellis upgrade` is a first-class command: channel-aware defaults (`latest` / `beta` / `rc`), `--tag`, `--dry-run`. Hardened: tag validation, no shell on POSIX, Windows `cmd.exe` invocation, troubleshooting output for npm / PATH failures.
- `trellis update` refreshes registry-backed `.trellis/spec` templates through the existing hash/conflict flow. `trellis init --template <id>` persists the spec source/template in `.trellis/config.yaml` so subsequent updates re-fetch it. Supports direct spec registries and marketplace-style registries (#324).
- Hook timeout defaults raised: 30s for `SessionStart`, 15s for per-prompt workflow injection across hook-based platforms.
- `add_session.py` and `task.py archive` no longer print a generic `git add .trellis && git commit` fallback when `.gitignore` excludes `.trellis/`. They stage only specific Trellis-owned paths and auto-retry with `git add -f -- <paths>` only on `ignored by` stderr.

**Bundled skills**
- `trellis-spec-bootstrap` (renamed from typoed `trellis-spec-bootstarp`, #296): bootstraps `.trellis/spec/` from the real codebase. Auto-deployed to every supported platform's skill root.
- `trellis-session-insight` (new): teaches AI when to reach for `trellis mem` for cross-session recall.
- Channel runtime bundled skills auto-discovered by AI agents on every platform.

**Migration & update flow**
- `0.6.0-beta.0` is the load-bearing breaking manifest for the 0.5.x → 0.6.0 jump (rename / delete chain documented in its own `migrationGuide`).
- `0.6.0-beta.23` ships a `rename-dir` migration that renames installed `trellis-spec-bootstarp/` → `trellis-spec-bootstrap/` across 13 platform skill roots. Missing roots silently skip.
- This `0.6.0.json` carries `migrations: []` because GA = rc.0; the breaking work is already absorbed by anyone who walked the beta chain. `recommendMigrate: true` still fires the `--migrate` guidance for users coming directly from 0.5.x.

**Bug fixes (cut window)**
- fix(agents): drop explicit `mcp__exa__*` tools from bundled `trellis-implement` and `trellis-check` agent defs. Claude Code silently skipped agent registration when an explicit MCP tool name failed to resolve, so users without the Exa MCP server installed had every Trellis sub-agent disappear from the dispatch list. `trellis-research` keeps external search by folding `mcp__exa__*` + `mcp__chrome-devtools__*` into a single `mcp__*` wildcard. OpenCode files unchanged (different permission syntax). (#302, rc.0)
- Earlier 0.6 beta-line bug fixes are documented in their per-beta changelogs.

**Out of scope at GA**
- Open feature requests (#193, #318, #320, #325, #326, …) deferred to v0.7 or later. v0.6 feature surface was frozen at rc.0.
```

### Length / readability checks for the changelog string

- All section headers are `**bold**` with no `#` / `##` — terminal-rendered, no Mintlify involved.
- No nested bullet lists (terminal-render-safe).
- Every backtick-quoted identifier is grep-able (`trellis mem`, `@mindfoldhq/trellis-core`, `channel.worker_guard.idle_timeout`).
- Issue numbers (#296, #301, #302, #323, #324) inline at the end of bullets, mirroring v0.5.0's `(#226)` style.
- One blank line between sections. Bullet block has no trailing blank before the next `**Header**`.
- When encoded as JSON, all newlines become `\n`; the string is one long line in the file. Total ≈ 3.8 KB.

## 5. Concrete `notes` string template

Three user cohorts must be addressed explicitly:

```
**Stable release.** Promoted from 0.6.0-rc.0 with no new src/ changes since.

**Users on 0.5.x** (e.g. `0.5.19`, the last 0.5 stable): run `trellis update --migrate`. The `--migrate` flag is REQUIRED — the breaking-change gate from `0.6.0-beta.0` fires when traversing the migration chain, and a separate `rename-dir` migration from `0.6.0-beta.23` renames the bundled `trellis-spec-bootstarp/` skill directory to `trellis-spec-bootstrap/` across every configured platform skill root (missing roots silently skip).

**Users on any 0.6.0 prerelease** (`beta.X` / `rc.0`): plain `trellis update` is a clean version bump — no `--migrate` needed because you already absorbed the breaking manifest when you adopted the beta line.

**Codex users**: the v0.6 cycle flipped `codex.dispatch_mode` default from `sub-agent` to `inline`. If you previously uncommented `dispatch_mode: sub-agent` in `.trellis/config.yaml`, that opt-in still routes to the legacy sub-agent dispatch flow; if you left the line commented (the default), behavior is now inline.

**Channel runtime is opt-in.** The single-agent workflow you know from 0.5 still works unchanged. `trellis channel` and the bundled `trellis-session-insight` skill are auto-discovered but only fire when the AI decides the task calls for them.

Install: `npm install -g @mindfoldhq/trellis`
```

### Notes string design rules (lifted from v0.5.0.json)

- Lead with `**Stable release.**` framing — single-sentence handshake that "this came from rc.0, no surprises."
- One labeled paragraph per user cohort, bold cohort label at start. v0.5.0 used "Users on 0.4.x" / "Users on any 0.5.0 prerelease" / "Codex users".
- The `--migrate` requirement is the SINGLE most important line; call out the underlying reason (breaking gate) so a user who reads it understands and doesn't suspect a bug.
- The `npm install -g` line at the very end, no `@latest` suffix, no version pin. Exactly mirrors v0.5.0.
- Newlines in JSON-encoded string: `\n\n` between paragraphs (single blank line in rendered output), `\n` for within-paragraph wraps where needed.

## Caveats / Not Found

- The exact rendering width of the terminal output during `trellis update` was not directly inspected — assumption is ~100 cols based on v0.5.0's section formatting that wraps cleanly at that width. If a reader has access to `inject-workflow-state.py` or the update-render path, double-check before final commit.
- This template does NOT include the `migrationGuide` field. If the reviewer wants to spell out a one-paragraph upgrade path on the manifest itself (rather than only in `notes`), the field is OPTIONAL for `breaking: false` manifests but accepted by the schema. Recommend leaving it omitted to match v0.5.0 exactly.
- The decision to set `recommendMigrate: true` is based on the v0.5.0 precedent and design.md §"Manifest field decisions." If the project has since changed the semantics of this flag (no evidence found), revisit.
- Final wording of the `changelog` lead sentence ("multi-agent channel runtime … you don't have to use them") is the author's call — keep, soften, or rewrite to match the GA changelog's tone in `docs-site/changelog/v0.6.0.mdx`.
